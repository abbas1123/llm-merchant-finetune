# llm-merchant-finetune

[![ci](https://github.com/abbas1123/llm-merchant-finetune/actions/workflows/ci.yml/badge.svg)](https://github.com/abbas1123/llm-merchant-finetune/actions/workflows/ci.yml)

Card statements butcher merchant names: `SQ *BEAN THEORY COFF`, `PP*BAYFRONT-PLAZA * 6593`, `NTBSKT.COM SAN ANTONIO TX`. This repo LoRA-fine-tunes Qwen2.5-0.5B-Instruct to turn those descriptors into structured JSON with a clean merchant name and one of 20 spending categories. It is a companion to my [realtime-fraud-ml](https://github.com/abbas1123/realtime-fraud-ml) project, where merchant identity and category are model features, and noisy descriptors are one of the main things that degrade them.

## Data

The dataset is synthetic and generated by a seeded script, so every run is reproducible. A pool of ~250 merchants across 20 MCC-style categories goes through the corruptions real acquirers apply: payment-processor prefixes (`SQ *`, `TST*`, `PAYPAL *`), vowel-stripped abbreviations, store numbers, city/state suffixes, reference digits and hard truncation at the statement field width. A few rows from the committed sample:

| raw | merchant_name | category |
|---|---|---|
| `tSt* PrEciSionpAinT&tiLe AUsTin TX` | Precision Paint & Tile | home_improvement |
| `Sptfy  STORE 715` | Spotify | streaming |
| `GOLDEN VAL GRCRS #68181.COM LOS ANGELES CA` | Golden Valley Grocers | grocery |
| `SQ *MET-TRANSIT-AUTH * 374057060` | Metro Transit Authority | transit |

Splits are 6,000 train / 1,000 val / 1,000 test with no raw string shared between splits. `data/sample.jsonl` is committed; the full splits are regenerated with one command.

## Training setup

| | |
|---|---|
| Base model | [Qwen/Qwen2.5-0.5B-Instruct](https://huggingface.co/Qwen/Qwen2.5-0.5B-Instruct) (Apache-2.0) |
| Method | LoRA (r=16, alpha=32, dropout=0.05) on all attention and MLP projections |
| Trainable params | 8.8M of 502.8M (1.75%) |
| Schedule | 2 epochs over 6,000 examples, batch 16, lr 2e-4 cosine, 20 warmup steps |
| Precision | bf16 |
| Hardware | NVIDIA GeForce RTX 5070 (12 GB) |
| Wall time | 20.5 minutes |

Training uses a plain `transformers` Trainer with the prompt tokens masked out of the loss (labels set to -100), so only the JSON completion is learned. I skipped TRL on purpose; for a single-turn SFT task the plain Trainer does the same job with one less fast-moving dependency. The adapter (a few MB) is committed under `models/adapter/` together with the training config and the loss curve, so evaluation and serving work without retraining.

## Results

Measured on the held-out 1,000-example test split, greedy decoding, single pass. A prediction only counts if the output parses as a JSON object with both fields; malformed output scores zero on every metric. Merchant names are compared after lowercasing and punctuation stripping.

| Metric | Base model (zero-shot) | LoRA fine-tuned | Delta |
|---|---|---|---|
| JSON validity | 100.0% | 100.0% | +0.0 pp |
| Category accuracy | 18.6% | 98.2% | +79.6 pp |
| Merchant name exact match | 16.9% | 97.2% | +80.3 pp |
| Both fields correct | 4.3% | 97.2% | +92.9 pp |

The instruct-tuned base already emits well-formed JSON on every input, so fine-tuning buys nothing on format. All of the gain is content: zero-shot, the model invents categories outside the taxonomy ("electricity", "transportation"), echoes raw uppercase fragments as merchant names and leaves store numbers in them; after two epochs of LoRA it has the 20-label taxonomy and the cleanup conventions down almost perfectly.

Full numbers and the exact eval config are in [eval/results.md](eval/results.md).

### Parsing robustness

Those accuracy numbers only hold if the parser reliably turns real, messy model
output into a clean prediction. `eval/run_robustness.py` stress-tests that layer
without the model or a GPU: it feeds prose-wrapped, markdown-fenced, extra-key,
whitespace-noisy and decoy-object outputs (which should still extract) alongside
single-quoted, truncated and wrong-typed outputs (which should be rejected, not
guessed).

| check | result |
|---|---|
| extraction on messy-but-recoverable outputs | 100% (7 cases) |
| safe rejection of malformed outputs | 100% (6 cases) |

Report in [eval/robustness_report.md](eval/robustness_report.md); reproduce with
`python eval/run_robustness.py`.

## Quickstart

```powershell
python -m venv .venv
.venv\Scripts\pip install -r requirements.txt   # on Linux: .venv/bin/pip

# 1. generate the dataset (seeded, ~10 seconds)
python -m merchant_llm.data.generate --out data

# 2. zero-shot baseline of the base model
python -m merchant_llm.baseline

# 3. LoRA fine-tune (skip if you use the committed adapter)
python -m merchant_llm.train

# 4. evaluate the tuned model and write the before/after report
python -m merchant_llm.evaluate

# 5. serve it
python -m uvicorn merchant_llm.serve.app:app --port 8000
```

```bash
curl -s -X POST http://127.0.0.1:8000/normalize \
  -H "Content-Type: application/json" \
  -d '{"raw": "SQ *BEAN THEORY COFF AUSTIN TX"}'
```

```json
{"merchant_name":"Bean Theory","category":"coffee","parsed":true,"latency_ms":317.6}
```

Measured over 30 sequential requests on the RTX 5070 (single stream, greedy decoding), median latency is 416 ms with p90 at 471 ms; nearly all of it is autoregressive decoding of the ~25-token JSON. `GET /healthz` reports the loaded model and device. Settings (base model, adapter path, device, token budget) come from `MERCHANT_`-prefixed environment variables via pydantic-settings.

The GPU wheel for torch is not on PyPI; see the comment at the top of `requirements.txt`. CPU works for inference and evaluation, just slower.

## Tests

```
pytest -q
```

CI (GitHub Actions) installs `requirements-ci.txt`, which has no torch or transformers, and runs ruff plus the pure-python suite: generator determinism and split disjointness, prompt formatting, the output parser and the scoring math. Tests that need model weights are marked `local` and only run with `MERCHANT_LOCAL_TESTS=1`.

## Limitations

- The data is synthetic. The corruption patterns imitate real statement formats, but the merchant pool is mostly invented and the noise distribution is mine. Numbers here will not transfer one-to-one to production descriptors; treat them as an upper bound on how much signal a small model can extract from this task shape.
- Splits are disjoint on raw strings, not on merchants. The model sees every merchant during training and is tested on unseen corruptions of known names. That mirrors the production case (the merchant dictionary is known) but it does not measure generalization to brand-new merchants.
- One model size (0.5B), one task, one language. No comparison against a cheaper non-LLM baseline like fuzzy matching over a merchant dictionary, which a production system should include.
- Merchant name matching is strict exact match after normalization, so near-misses ("Golden Valley Grocer" vs "Golden Valley Grocers") count as failures.
- The 20-category taxonomy is flat and much smaller than the real MCC code space.

## Roadmap

- Evaluate on a slice of real (anonymized) descriptors to measure the synthetic-to-real gap
- Constrained JSON decoding at inference time instead of trusting the model to close its braces
- A fuzzy-match dictionary baseline to quantify what the LLM actually adds
- Quantized (int8/int4) CPU inference to cut serving cost
